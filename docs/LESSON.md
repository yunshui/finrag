# LESSON.md — 经验总结文档

## 1. 项目回顾

finrag 项目从零开始，基于已有的 API 示例（curl 命令 + JSON 响应），在单一会话中完成了：
- 项目结构搭建
- 核心 CLI 工具实现（204 行 Python）
- 中英文双语文档
- 8 份规范化文档
- Git 提交并推送至 GitHub

## 2. 做得好的方面

### 2.1 单文件架构

选择将所有功能放入单个 `finrag.py` 文件是正确的决定。原因：
- 当前功能范围明确且有限
- 单文件便于分发（用户只需拷贝一个文件 + 一个配置）
- 模块通过注释分隔（`# ── Config ──`），代码组织清晰
- 避免了过度工程化（如创建包、`__init__.py`、多文件导入）

### 2.2 配置与代码分离

通过 `conf/setting.json` 管理默认参数：
- 用户修改配置无需改动代码
- CLI 参数可覆盖任意配置项，灵活性高
- JSON 格式天然可读，Python 原生支持

### 2.3 完善的日志系统

- 同时输出到控制台和文件，兼顾即时反馈和事后追溯
- 按天轮转日志文件，避免单文件过大
- 结构化日志格式（` | ` 分隔字段），便于 grep 和脚本解析
- 记录完整的请求生命周期（开始 → 重试 → 成功/失败）

### 2.4 安全的文件冲突处理

使用随机 5 字符后缀（60M+ 组合）而非覆盖已有文件：
- 防止意外丢失之前的输出结果
- 后缀长度和字符集平衡了唯一性和可读性

### 2.5 从第一行代码就开始版本控制

- 先创建文件，立即提交
- 后续每个功能步骤独立提交
- 提交信息遵循约定式提交格式（feat:, docs:）

## 3. 可以改进的方面

### 3.1 缺少 .gitignore

应在项目初始化时创建 `.gitignore`，排除：
```
logs/
output/
conf/setting.json
__pycache__/
*.pyc
.DS_Store
```

特别是 `conf/setting.json` 可能包含敏感信息（如自定义 `client_id`），不应提交到仓库。

### 3.2 缺少单元测试

当前没有任何测试。建议至少添加：
- `extract_line_numbers()` 的正则匹配测试
- `resolve_output_path()` 的冲突处理测试
- `build_config()` 的 CLI 覆盖测试

### 3.3 配置验证缺失

没有验证 `conf/setting.json` 中必填字段的存在性和类型。如果缺少 `api_url` 或 `query`，会在运行时产生不太友好的错误信息。

### 3.4 错误处理可以细化

- `requests.post()` 成功后 `resp.json()` 可能抛出 `json.JSONDecodeError`，当前未捕获
- 没有检查响应 JSON 是否包含预期的 `context` 字段
- 配置文件的 `FileNotFoundError` 直接抛出 Python 异常，未转换为友好错误信息

### 3.5 缺少进度指示

对于耗时较长的请求（默认超时 60 秒），用户只能看到日志中的开始信息，之后没有任何中间状态反馈。可以考虑：
- 请求期间打印一个旋转指示器（`/` `-` `\` `|`）
- 使用 `tqdm` 或类似库显示进度

### 3.6 随机数生成

`random.choices()` 不是密码学安全的。虽然对于文件名冲突场景足够，但如果需要更强的唯一性保证，可以使用 `secrets.token_hex(3)`。

## 4. 关键决策记录

### 4.1 为什么选择线性退避而非指数退避？

**决策**：使用 `1s × attempt` 的线性退避（1s, 2s, 3s）。

**理由**：
- API 调用场景下，网络问题通常很快恢复
- 指数退避（1s, 2s, 4s, 8s）在重试 3 次时总等待时间为 15s，线性为 6s
- 默认超时 60s × 3 次重试已经很长，退避时间不宜过长

### 4.2 为什么只提取行号而不提取章节？

**决策**：仅使用正则 `来源:\s*行\s*(\d+)` 提取行号。

**理由**：
- 行号是最稳定的标识符
- 章节标题格式不统一（有的包含、有的不包含）
- 行号可用于回溯到原始文档的精确位置

### 4.3 为什么使用 `requests` 而非 `urllib`？

**决策**：引入唯一外部依赖 `requests`。

**理由**：
- `requests` 的 API 更简洁（`files`, `data`, `headers` 参数清晰）
- 内置连接池和更好的错误处理
- `urllib` 的 `MultipartEncoder` 实现较为繁琐
- 对于生产工具，多一个成熟可靠的依赖是可接受的

## 5. 通用经验

### 5.1 从 API 示例反推客户端设计

当只有 curl 示例和 JSON 输出时：
1. 先完整理解 API 的请求格式和响应结构
2. 提取关键数据流（哪些字段是输入、哪些是输出）
3. 识别需要自动化的痛点（如提取 `context`、重试、日志）
4. 设计最小可用产品（MVP），再逐步增强

### 5.2 文档先行

在编写代码的同时编写文档，而非事后补充：
- PRD 帮助明确功能需求边界
- TECH.md 帮助梳理模块设计和依赖关系
- APP_FLOW.md 帮助验证流程的完整性
- 文档中的表格直接转化为代码中的注释和结构

### 5.3 测试应该在开发过程中进行

本项目先完成了全部代码，然后计划测试。更好的做法是：
- 每完成一个模块就测试该模块
- 例如 `extract_line_numbers()` 可以用示例 JSON 中的 context 字符串立即验证
